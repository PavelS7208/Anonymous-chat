## Реализация Этап 4

### Результаты собственного ревью этапа 3

1. В доменном объекте комната (Room + Member) зашиты детали реализации стриминга через каналы, буферы, алгоритмя отработки с медленными клиентами и т.п. По хорошему, это нужно вынести на инфораструктурный слой, чтоюы при тестировании оставить только логику без необходимости мокать инфраструктурные сущности механизма передачи. Подумать как это реализовать, например что-то похожее на Pub-Sub. Оценить стоит ли, плюсы-минусы.
2. В основных методах сервисного слоя Join, Stream, Send реализация состояит из последовательности (строго в определенном порядке) вызовов необходимых методов. При этом уже 5+ таких шагов и скорее всего будет расширяться. Подумать, может стоит перейти на паттерн pipeline для упрощения понимания и добавления (в случае необходимости) новых составляющих алгоритма работы чата.
3. Защитники от сетевых атак добавлялись произвольно по мере понимания их необходммости, они с разной архитектурой. Вызовы методов с разной сигнатурой, разным неймингом и т.п. Подумать, как бы их "спрятать" за один (несколько) универсальных интерфейсов с унифицированной сигнатурой методов.
4. Механизм пометок зависших или давно не используемых комнат и удаления их реализован в обном типе что и InMemory хранилище комнат, как дополнительные методы. Подумать может стоит вынести, для чистоты ответвенности, данный  в отдельный сервис за типовой интерфейс.
5. Юнит тесты покрытие увеличить. Рассмотреть инфраструктурные - разобраться как их сделать.

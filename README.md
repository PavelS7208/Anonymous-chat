## Реализация Этап 4

### Результаты собственного ревью этапа 3

1. В доменном объекте комната (Room + Member) зашиты детали реализации стриминга через каналы, буферы, алгоритмя отработки с медленными клиентами и т.п. По хорошему, это нужно вынести на инфраструктурный слой, чтобы при тестировании оставить только логику без необходимости мокать инфраструктурные сущности механизма передачи. Подумать как это реализовать, например что-то похожее на Pub-Sub. Оценить стоит ли, плюсы-минусы. **(Выполнено)**
2. В основных методах сервисного слоя Join, Stream, Send реализация состояит из последовательности (строго в определенном порядке) вызовов необходимых методов. При этом уже 5+ таких шагов и скорее всего будет расширяться. Подумать, может стоит перейти на паттерн pipeline для упрощения понимания и добавления (в случае необходимости) новых составляющих алгоритма работы чата.
3. Защитники от сетевых атак добавлялись произвольно по мере понимания их необходммости, они с разной архитектурой. Вызовы методов с разной сигнатурой, разным неймингом и т.п. Подумать, как бы их "спрятать" за один (несколько) универсальных интерфейсов с унифицированной сигнатурой методов.
4. Механизм пометок зависших или давно не используемых комнат и удаления их реализован в одном общем типе что и InMemory хранилище комнат, как дополнительные методы. Подумать может стоит вынести, для чистоты ответвенности, данный функционал в отдельный сервис за типовой интерфейс.
5. В ЖЦ чата интерфейс содержит два метода Start/ Close. Похоже требуется что-то вроде Wait() где на основе WaitGroup или иного ожидается завершение запущенных сервисных процедур. Сейчас приходится в доменных объектах и сервисном слое проверять на isClose() чтоюбы не попасть на ситуацию, когда ассинхронность Close при завершении сервера или иных причинах идет в неконтролируемом порядке.
6. В текущей архитектуру жив ли клиент, мы узнаем только при отправке ему сообщения. Подумать - может стоить придумать механизм определения состояния клиента и при кратковременных сетевых проблемах переподолючение (с защитой взлома) без новой регистрации с новой парой ключей.
7. Ошибки уровня транспорта и маппинг на них доменных и сервисных ошибок "размазан" по коду. Подумать как собрать это все в одном месте, для простоты настройки/изменения какие коды и с каким текстом и доп заголовками отдаем по http
8. Юнит тесты покрытие увеличить. Рассмотреть инфраструктурные - разобраться как их сделать.
